É um pequeno servidor embutido no próprio Power BI Desktop. Ele deixa scripts e agentes de IA conversarem com o arquivo aberto sem precisar de clique nenhum.

contexto · protocolo padrão da Microsoft

Power BI Modeling MCP

Servidor MCP mantido pela própria Microsoft — expõe o modelo semântico (TOM) do Power BI Desktop vivo: tabelas, medidas, relacionamentos, direto ao agente de IA.

Pra que serve

O agente lê e edita o modelo sem tocar em TMDL na mão. Toda mudança entra direto no modelo vivo — sem risco do Desktop sobrescrever a edição ao salvar.

Por que este, não o da extensão VS Code

Mesmo pacote (@microsoft/powerbi-modeling-mcp), mas registrado no escopo do terminal/CLI: funciona sem abrir o VS Code e evita duas instâncias do servidor rodando ao mesmo tempo.

fonteServidor oficial da Microsoft, em Public Preview: github.com/microsoft/powerbi-modeling-mcp
contexto · formato de arquivo

Por dentro do formato PBIP

Ao salvar como PBIP, o Power BI não grava um arquivo único — ele monta uma pasta inteira de projeto. A parte de relatório, dentro dela, é o que chamamos de PBIR.

PBIP o pacote completo

A pasta-mãe do projeto: reúne relatório e modelo de dados num único lugar.

PBIR o relatório em JSON

Páginas, visuais e tema do relatório, escritos como JSON dentro da pasta PBIP. É essa parte que a skill de authoring mexe.

COMO A PASTA FICA DEPOIS DE SALVAR

MeuRelatorio.pbip // arquivo raiz
MeuRelatorio.Report/ ← é o PBIR
  definition/
    report.json // config visual, tema
    pages/ ... page.json // uma pasta por página
      visuals/ ... visual.json // tipo, posição
MeuRelatorio.SemanticModel/ ← onde moram os dados
  definition/ ... *.tmdl // tabelas, medidas
  relationships, model // como se conectam
analogiaPense assim: PBIP é a caixa; PBIR é só um dos itens guardados dentro dela.
escopoA skill powerbi-report-authoring trabalha só dentro da pasta .Report. O .SemanticModel — onde ficam os dados — não é dela; isso é trabalho do Modeling MCP.
a peça-chave · formato de arquivo

Por que salvar em PBIP muda tudo

O Bridge só automatiza o que existe como texto no disco — e isso só acontece no formato PBIP.

⊘

.pbix

Formato fechado e binário. Não tem como editar o conteúdo por fora — o ciclo automático simplesmente não roda.

◉

.pbip

Uma pasta cheia de JSON, um arquivo por página e visual. Scripts e agentes editam direto — é o que o Bridge recarrega.

4 skills que dão superpoderes à IA

Skill é conhecimento especializado empacotado — passo a passo, convenção e regra de um domínio específico que a IA sozinha não tem de fábrica. Sem isso, o agente até tenta editar o PBIR ou montar o plano, mas no achismo; com a skill certa, ele segue sempre o mesmo processo.

powerbi-report-design

Cuida do design antes de qualquer arquivo existir: arquétipo de página, paleta, tipo de gráfico e tema.

→ gera: brief de design (só documento)

powerbi-report-authoring

Escreve e valida o JSON do relatório em si: páginas, visuais, filtros e tema, tudo dentro do PBIR.

→ gera: PBIR validado

powerbi-report-planning

O maestro: levanta requisito, inspeciona o modelo existente, monta o brief e só então aciona o build.

→ gera: requirements.md · plan.md

powerbi-report-management

Cuida do ciclo de vida no workspace: publica, baixa ou apaga relatório via API REST do Fabric.

→ gera: relatório publicado no Fabric

Precisa de Fabric + login az

semantic-model-authoring

Cria e edita o modelo por trás do relatório: tabelas, colunas, relacionamentos e, principalmente, medidas DAX — direto no TMDL ou via powerbi-modeling-mcp.

→ gera: modelo semântico validado (TMDL)

Opção 1 — Bridge: descoberta e plano

Você é engenheiro Power BI. Execute até o fim sem pedir confirmação. Contexto: existe um projeto .pbip nesta pasta (ou no caminho que o usuário informar), com um modelo e dataset desconhecidos — descubra tudo agora, não assuma nomes de tabela/coluna de nenhum outro projeto.

═══ PASSO 0 — PRÉ-REQUISITOS ═══
1. Confirme que a skill semantic-model-authoring está disponível (invocável por nome, ou os arquivos SKILL.md/references/ estão localizáveis). Se não estiver, PARE e diga: "Instale a pasta skills-for-fabric/skills no escopo global antes de continuar."
2. powerbi-report-author --version → se faltar: PARE e instrua a instalação.
3. Pule a checagem de atualização da skill (check-updates) — não é necessária para este fluxo de 2 prompts.

═══ GUARDRAIL LGPD (obrigatório) ═══
Leia SOMENTE arquivos de definição: *.tmdl e *.json dentro de <nome>.SemanticModel/definition/ e <nome>.Report/definition/. NUNCA abra arquivos de dados (CSV, Excel, parquet, cache.abf etc.) e nunca execute queries. Dados brutos podem conter dados pessoais e não devem entrar no contexto da IA.

═══ DESCOBERTA (via skill semantic-model-authoring) ═══
Invoque a skill semantic-model-authoring. Mesmo que powerbi-modeling-mcp esteja registrado, IGNORE-O — force o caminho Tier 2 (TMDL direto) descrito na skill, lendo os arquivos .tmdl em <nome>.SemanticModel/definition/ (tables/*.tmdl, relationships.tmdl). Nunca use MCP neste fluxo.

1. Extraia dos .tmdl: tabelas, colunas (nome + dataType), medidas existentes, relacionamentos.
2. Resuma: tabela(s) fato, dimensões, cardinalidade, medidas já existentes. Use SOMENTE nomes lidos dos arquivos — não invente.

═══ PROPOSTA (não crie nada ainda) ═══
3. Sugira exatamente 2 visuais nativos, escolhidos a partir do schema real:
   - 1 card KPI: qual medida e por quê.
   - 1 gráfico de barras: qual medida × qual categoria e por quê.
4. Para cada visual, defina a(s) medida(s) necessária(s) com: name, expression DAX, formatString conforme a natureza (moeda "R$ #,##0.00" | contagem "#,##0" | ratio "0.0%"), displayFolder. Restrições (não haverá validação DAX antes de abrir o Desktop):
   - só agregações simples: SUM / AVERAGE / COUNT / DIVIDE;
   - nomes de tabela e coluna copiados EXATAMENTE dos .tmdl;
   - name e displayFolder das medidas novas em ASCII puro, SEM acento (ex.: "Salario Medio") — o TMDL será editado à mão e acento com encoding errado corrompe o arquivo (ver regra de encoding no Prompt 2). Nomes com acento ficam nos títulos/labels dos visuais;
   - se já existir medida equivalente, reuse — não proponha duplicata.

═══ HANDOFF ═══
5. Grave o plano em `_status/plano.md` na pasta do .pbip: schema resumido, nome exato da tabela alvo e do arquivo .tmdl dela, os 2 visuais e as medidas propostas (name, expression, formatString, displayFolder). Esse arquivo é a única memória do prompt de execução.
6. Mostre o plano em tabela resumida e finalize com: "Plano gravado em _status/plano.md. FECHE o Power BI Desktop e rode o prompt 2."

Opção 1 — Bridge: execução completa

Você é engenheiro Power BI. Execute até o fim sem pedir confirmação. Pare apenas se faltar pré-requisito.

═══ REGRA CRÍTICA Nº 1 — DESKTOP FECHADO ═══
O Power BI Desktop deve estar FECHADO: aberto, ele mantém o modelo em memória e sobrescreve qualquer edição de TMDL feita em disco. Verificação: powerbi-desktop status → se retornar instância ativa / reportDir preenchido, PARE e peça pro usuário fechar o Desktop. Se o CLI powerbi-desktop não existir na máquina, peça confirmação explícita do usuário de que o Desktop está fechado antes de continuar.

═══ CONTEXTO ═══
Leia `_status/plano.md` na pasta do .pbip (gerado pelo prompt 1). Se não existir e o plano não estiver nesta sessão: PARE e peça pra rodar o prompt 1.

═══ PRÉ-REQUISITOS ═══
- powerbi-report-author --version → se faltar: PARE e instrua a instalação.
- Confirme acesso às skills semantic-model-authoring e powerbi-report-authoring (por nome ou pelos arquivos .md). Sem elas, as regras abaixo são a única rede de segurança — não pule nenhuma.

═══ GUARDRAIL LGPD ═══
Só arquivos de definição (*.tmdl, *.json). Nunca arquivos de dados.

═══ FASE 1 — MEDIDAS (edição direta de TMDL, via skill semantic-model-authoring) ═══

Invoque a skill semantic-model-authoring, caminho Tier 2 (TMDL direto — sem MCP, mesmo se registrado). Leia tmdl-guidelines-pt-BR.md antes de editar qualquer .tmdl — ela cobre nomenclatura e formato de measure. Regra de encoding abaixo é ADICIONAL a ela (rede de segurança extra deste fluxo, por rodar em máquinas de terceiros sem supervisão):

REGRA DE ENCODING (obrigatória — TMDL exige UTF-8; gravação errada corrompe o arquivo e o Desktop não abre: "Não é possível converter bytes [E9] ... para Unicode"):
a) NUNCA escreva/edite .tmdl via shell (Set-Content, Out-File, echo >, sed, heredoc) — pode gravar em ANSI. Use SOMENTE a ferramenta de edição de arquivo nativa do seu ambiente (a que preserva UTF-8).
b) Todo texto que VOCÊ inserir no .tmdl deve ser ASCII puro — SEM acentos em name, descrição (///) e displayFolder (ex.: "Salario Medio", não "Salário Médio"). Nomes bonitos com acento vão nos títulos/labels dos visuais (Fase 2), nunca no TMDL. Acentos já existentes no arquivo (gravados pelo Desktop) ficam como estão — não os toque.
c) Após CADA edição de .tmdl, valide o encoding antes de continuar: powershell -Command "[IO.File]::ReadAllText('<arquivo>.tmdl',[Text.UTF8Encoding]::new($false,$true))>$null;'UTF-8 OK'" (ou equivalente no seu SO). Se lançar erro de conversão, o arquivo está corrompido: reescreva-o inteiro em UTF-8 e valide de novo. Não avance sem "UTF-8 OK".

1. Abra o .tmdl da tabela alvo (caminho no plano). Liste as medidas existentes — NÃO crie duplicata (o Desktop rejeita: "TMDL objects cannot be merged").
2. Insira cada medida do plano ANTES das colunas, no formato:

   /// Descricao da medida (ASCII, sem acento)
   measure 'Nome' = EXPRESSAO_DAX
     formatString: <formato>
     displayFolder: <pasta>

3. Siga EXATAMENTE o padrão de indentação do arquivo existente (TMDL é sensível a indentação). Revise cada expression contra os nomes exatos de coluna do arquivo — é a única validação disponível.

═══ FASE 2 — VISUAIS (via skill powerbi-report-authoring) ═══

Invoque a skill powerbi-report-authoring. Antes de escrever qualquer visual.json, leia references/authoring.md (seção "Erros Comuns") e references/formatting.md — cobrem, entre outros, os padrões abaixo que já derrubaram builds anteriores. Não reproduza esses erros:

- `field` sem wrapper `Column`/`Measure`/`Aggregation` (o Desktop não abre o relatório).
- `objects.*` como objeto solto em vez de array `[{ "properties": {...} }]`.
- Literais malformados: `"14pt"`, número sem sufixo `D`/`L`, cor como string crua em vez de `{"solid":{"color":{"expr":{"Literal":{"Value":"'#RRGGBB'"}}}}}`. Gere cada literal via `powerbi-report-author expr encode <valor> --kind <bool|number|integer|string|color>`.
- `filterConfig` na RAIZ do visual.json (irmão de `visual`), nunca dentro de `visual`.
- `position` com números diretos, nunca envolto em `{"expr":{"Literal":...}}`.
- Se `powerbi-report-author validate` reportar `PBIR_SCHEMA_UNREACHABLE`, a checagem de schema foi PULADA — não é sucesso. Siga o procedimento da skill (SKILL.md § Validation result handling): apontar temporariamente `$schema` pra última versão publicada, revalidar, corrigir, restaurar.

Alvo: primeira página do relatório (crie se não houver).

Layout (canvas 1920×1080, grid 8px, margem 32):
- Card KPI:  x=32  y=32   w=360  h=200
- Barras:    x=32  y=264  w=920  h=480 — horizontal se for ranking, baseline zero sempre, ordenado descendente.

Nomes legíveis: título do visual = pergunta analítica; eixos e labels com display names em linguagem natural, nunca nome técnico de coluna.

Grave arquivos do PBIR SOMENTE via ferramenta de edição de arquivo nativa (nunca via shell), sempre UTF-8 — mesma regra de encoding da Fase 1.

═══ FASE 3 — FECHAMENTO ═══
1. powerbi-report-author validate (1 vez). Warning de "htmlContent" é ignorável; PBIR_SCHEMA_UNREACHABLE exige o procedimento da Fase 2 antes de concluir.
2. Instrua o usuário: "Abra o .pbip no Power BI Desktop (File → Open) para conferir. Se alguma medida der erro, me envie o texto exato que eu corrijo."
3. Opcional: depois que o usuário confirmar o Desktop aberto, rode powerbi-desktop screenshot para verificação visual.

═══ PASSO 0 — PRÉ-REQUISITOS ═══
1. Conecte ao modelo via powerbi-modeling-mcp (connection_operations → instância local do Desktop). Se não conectar: PARE e instrua o usuário a (1) abrir o .pbip no Power BI Desktop e (2) confirmar que o powerbi-modeling-mcp está registrado. Não tente fallback editando TMDL na mão.
2. Confirme que a skill semantic-model-authoring está disponível (invocável por nome, ou os arquivos SKILL.md/references/ estão localizáveis). Se não estiver, avise o usuário e prossiga só com o MCP — as regras deste prompt cobrem o essencial, mas sem a skill perde-se a rede de segurança extra.
3. Pule a checagem de atualização da skill (check-updates) — não é necessária para este fluxo de 2 prompts.

═══ GUARDRAIL LGPD (obrigatório) ═══
- Use SOMENTE queries de metadata: INFO.TABLES(), INFO.COLUMNS(), INFO.MEASURES(), INFO.RELATIONSHIPS().
- NUNCA execute EVALUATE sobre tabelas ou colunas do modelo (nada de amostrar linhas). Dados brutos podem conter dados pessoais e não devem entrar no contexto da IA.

═══ DESCOBERTA ═══
1. Via dax_query_operations + INFO.*, liste: tabelas, colunas (com tipo), medidas existentes, relacionamentos.
2. Resuma: tabela(s) fato, dimensões, cardinalidade dos relacionamentos, medidas já existentes. Use SOMENTE nomes retornados — não invente.

═══ PROPOSTA (não crie nada ainda) ═══
3. Sugira exatamente 2 visuais nativos, escolhidos a partir do schema real:
   - 1 card KPI: qual medida e por quê.
   - 1 gráfico de barras: qual medida × qual categoria e por quê.
4. Para cada visual, defina a(s) medida(s) necessária(s) com: name, expression DAX (só agregações simples: SUM / AVERAGE / COUNT / DIVIDE), formatString conforme a natureza (moeda "R$ #,##0.00" | contagem "#,##0" | ratio "0.0%"), displayFolder. Se já existir medida equivalente no modelo, reuse — não proponha duplicata.

═══ HANDOFF ═══
5. Grave o plano em `_status/plano.md` na pasta do .pbip: schema resumido, nome exato da tabela alvo, os 2 visuais e as medidas propostas (name, expression, formatString, displayFolder). Esse arquivo é a única memória do prompt de execução.

═══ FASE 2 — VISUAIS (via skill powerbi-report-authoring) ═══

Invoque a skill powerbi-report-authoring. Antes de escrever qualquer visual.json, leia references/authoring.md (seção "Erros Comuns") e references/formatting.md — cobrem, entre outros, os padrões abaixo que já derrubaram builds anteriores. Não reproduza esses erros:

- `field` sem wrapper `Column`/`Measure`/`Aggregation` (o Desktop não abre o relatório).
