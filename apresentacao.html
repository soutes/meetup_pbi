É um pequeno servidor embutido no próprio Power BI Desktop. Ele deixa scripts e agentes de IA conversarem com o arquivo aberto sem precisar de clique nenhum.

contexto · protocolo padrão da Microsoft

Power BI Modeling MCP

Servidor MCP mantido pela própria Microsoft — expõe o modelo semântico (TOM) do Power BI Desktop vivo: tabelas, medidas, relacionamentos, direto ao agente de IA.

Pra que serve

O agente lê e edita o modelo sem tocar em TMDL na mão. Toda mudança entra direto no modelo vivo — sem risco do Desktop sobrescrever a edição ao salvar.

Por que este, não o da extensão VS Code

Mesmo pacote (@microsoft/powerbi-modeling-mcp), mas registrado no escopo do terminal/CLI: funciona sem abrir o VS Code e evita duas instâncias do servidor rodando ao mesmo tempo.

fonteServidor oficial da Microsoft, em Public Preview: github.com/microsoft/powerbi-modeling-mcp
contexto · formato de arquivo

Por dentro do formato PBIP

Ao salvar como PBIP, o Power BI não grava um arquivo único — ele monta uma pasta inteira de projeto. A parte de relatório, dentro dela, é o que chamamos de PBIR.

PBIP o pacote completo

A pasta-mãe do projeto: reúne relatório e modelo de dados num único lugar.

PBIR o relatório em JSON

Páginas, visuais e tema do relatório, escritos como JSON dentro da pasta PBIP. É essa parte que a skill de authoring mexe.

COMO A PASTA FICA DEPOIS DE SALVAR

MeuRelatorio.pbip // arquivo raiz
MeuRelatorio.Report/ ← é o PBIR
  definition/
    report.json // config visual, tema
    pages/ ... page.json // uma pasta por página
      visuals/ ... visual.json // tipo, posição
MeuRelatorio.SemanticModel/ ← onde moram os dados
  definition/ ... *.tmdl // tabelas, medidas
  relationships, model // como se conectam
analogiaPense assim: PBIP é a caixa; PBIR é só um dos itens guardados dentro dela.
escopoA skill powerbi-report-authoring trabalha só dentro da pasta .Report. O .SemanticModel — onde ficam os dados — não é dela; isso é trabalho do Modeling MCP.
a peça-chave · formato de arquivo

Por que salvar em PBIP muda tudo

O Bridge só automatiza o que existe como texto no disco — e isso só acontece no formato PBIP.

⊘

.pbix

Formato fechado e binário. Não tem como editar o conteúdo por fora — o ciclo automático simplesmente não roda.

◉

.pbip

Uma pasta cheia de JSON, um arquivo por página e visual. Scripts e agentes editam direto — é o que o Bridge recarrega.

as 4 skills · o que roda 100% local

4 skills que dão superpoderes à IA

Skill é conhecimento especializado empacotado — passo a passo, convenção e regra de um domínio específico que a IA sozinha não tem de fábrica. Sem isso, o agente até tenta editar o PBIR ou montar o plano, mas no achismo; com a skill certa, ele segue sempre o mesmo processo.

powerbi-report-design

Cuida do design antes de qualquer arquivo existir: arquétipo de página, paleta, tipo de gráfico e tema.

→ gera: brief de design (só documento)

100% local · não grava nada em disco

powerbi-report-authoring

Escreve e valida o JSON do relatório em si: páginas, visuais, filtros e tema, tudo dentro do PBIR.

→ gera: PBIR validado

100% local · precisa do Desktop aberto

powerbi-report-planning

O maestro: levanta requisito, inspeciona o modelo existente, monta o brief e só então aciona o build.

→ gera: requirements.md · plan.md

Local · inspeção passa pelo Modeling MCP

powerbi-report-management

Cuida do ciclo de vida no workspace: publica, baixa ou apaga relatório via API REST do Fabric.

→ gera: relatório publicado no Fabric

Precisa de Fabric + login az

semantic-model-authoring

Cria e edita o modelo por trás do relatório: tabelas, colunas, relacionamentos e, principalmente, medidas DAX — direto no TMDL ou via powerbi-modeling-mcp.

→ gera: modelo semântico validado (TMDL)

Local · TMDL direto (Desktop fechado) ou MCP ao vivo (Desktop aberto)

E os dados, ela vê?

Enxerga o layout: páginas, visuais e quais campos cada gráfico usa.

Não enxerga o conteúdo das tabelas. O modelo semântico está fora do alcance dela.

É o limite dessa skill: só camada visual. Pra ler dado de verdade, quem entra em campo é o Modeling MCP.

ao vivo · opção 1 · bridge (sem mcp) · prompt 1 de 2 · descoberta e plano

Opção 1 — Bridge: modelo mapeado direto do TMDL

Sem MCP: leitura direta dos arquivos .tmdl. O Power BI Desktop fica fechado desde este primeiro prompt — no fluxo via Bridge, vamos criar as medidas e visuais diretamente no PBIR. Power BI Desktop só abre quando concluir a fase 2 do prompt.

Você é engenheiro Power BI. Execute até o fim sem pedir confirmação. Existe um projeto .pbip nesta pasta, com modelo e dataset desconhecidos — descubra tudo agora, não assuma nomes de nenhum outro projeto.

Invoque a skill /semantic-model-authoring, caminho TMDL direto (Tier 2) — NUNCA use powerbi-modeling-mcp neste fluxo, mesmo que esteja registrado. Power BI Desktop deve estar FECHADO: aberto, ele sobrescreve qualquer edição de TMDL feita em disco.

Leia SOMENTE arquivos de definição (*.tmdl, *.json em <nome>.SemanticModel/definition/ e <nome>.Report/definition/) — nunca CSV, Excel, cache.abf ou qualquer engine de query. Dados de linha não entram no contexto da IA.

Liste tabelas, colunas (com tipo), medidas existentes e relacionamentos. Resuma tabela(s) fato, dimensões, cardinalidade. Use só nomes lidos dos arquivos — não invente.

Proponha exatamente 2 visuais nativos — 1 card KPI e 1 gráfico de barras — com as medidas necessárias: name, expression DAX, formatString conforme a natureza (moeda "R$ #,##0.00" | contagem "#,##0" | ratio "0.0%"), displayFolder. Nomes de medida em ASCII puro, sem acento — o TMDL vai ser editado à mão no próximo prompt e acento com encoding errado corrompe o arquivo. Se já existir medida equivalente, reuse.

Grave o plano em _status/plano.md: schema resumido, tabela e arquivo .tmdl alvo, os 2 visuais e as medidas propostas. Mostre o plano e pare — a criação vem no próximo prompt.
formatoDenso e direto: a skill certa já invocada com /skill, contexto completo, zero ida-e-volta.
ao vivo · opção 1 · bridge (sem mcp) · prompt 2 de 2 · visuais

Opção 1 — Bridge: visuais em PBIR

Continuação — Desktop continua fechado até o fim. Só reabre depois de tudo salvo, pra conferir.

Invoque a skill /powerbi-report-authoring. Alvo: primeira página do relatório (crie se não houver). Card KPI: x=32 y=32 w=360 h=200. Barras: x=32 y=264 w=920 h=480 — baseline zero, ordenado descendente. Títulos = pergunta analítica; eixos e labels em linguagem natural, nunca nome técnico de coluna.

Antes de escrever visual.json, leia authoring.md § Erros Comuns e formatting.md. Não repita: field sem wrapper Column/Measure/Aggregation (Desktop não abre o relatório); objects.* como objeto solto — sempre array [{ "properties": {...} }]; literal sem sufixo D/L ("14pt", "8" cru), cor como string crua; filterConfig dentro de "visual" (mora na raiz, irmão de "visual"); position embrulhado em Literal (usa número direto: "x": 32). Gere cada literal com powerbi-report-author expr encode <valor> --kind <bool|number|integer|string|color>.

FECHAMENTO: powerbi-report-author validate — 0 erros. PBIR_SCHEMA_UNREACHABLE significa que a checagem de schema foi PULADA, não é sucesso: aponte $schema temporariamente pra última versão publicada, revalide, corrija, restaure. Instrua o usuário: "Abra o .pbip no Power BI Desktop pra conferir."

ao vivo · opção 2 · mcp ao vivo · prompt 2 de 2 · fase 1 · medidas

Opção 2 — MCP: medida validada na hora

Via powerbi-modeling-mcp: cada medida é criada e validada na hora antes de seguir pra próxima — sem esperar até o fim pra descobrir um erro.

Continuação: plano já está em _status/plano.md. Desktop segue ABERTO. Execute até o fim, sem pedir confirmação.

Invoque a skill /semantic-model-authoring, via powerbi-modeling-mcp. Liste as medidas existentes (INFO.MEASURES()). NÃO crie duplicata. Para cada medida do plano que ainda não existe: measure_operations → Create { tableName, name, expression, formatString, displayFolder }, depois dax_query_operations → EVALUATE ROW("t",[<medida>]). Erro → mostre o texto exato, corrija, revalide. Não avance com erro.

Ao final de TODAS: database_operations → ExportToTmdlFolder (uma vez só, não por medida — grava o TMDL em UTF-8 correto).
