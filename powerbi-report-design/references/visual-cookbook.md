### Theme vs Per-Visual — What Goes Where

The report theme should handle these defaults automatically:
- Table/matrix: row banding, header styling, grow-to-fit
- Bar/column: data labels on, value axis hidden, gridlines hidden, bar borders off
- Card: bold values, show labels, tight spacing
- Line chart: smooth curves, shaded area, sparse labels
- Slicer: header/items font sizing
- Global: hidden visual headers, hidden drop shadows

**Theme/per-visual boundary** — some visual treatments are part of the design
but may require per-visual implementation rather than theme-only changes.
Record these requirements in the brief; `powerbi-report-authoring` owns the exact
PBIR JSON mechanics.

| Design requirement | Affects | Why call it out |
|---|---|---|
| Rounded container corners | Visual containers | Needed for card-like layouts or softer tones |
| Explicit card/container background | Visual layering | Needed when page and visual surfaces differ |
| Padding / internal spacing | Visual containers and cards | Needed when density or whitespace is part of the identity |
| Chart/table title and subtitle state | Charts and tables | Needed so visuals have deliberate titles without noisy autogenerated field-name subtitles |
| Card accent bars / outlines / layout | KPI cards | Card identity is usually per-metric |
| KPI value color and label text | KPI cards | Values and labels are content-specific |
| Bar sort order | Bar/column charts | Sorting carries analytical meaning |
| Data-color gradients or highlighted categories | Bars, matrices, heatmaps | Colors depend on the specific measure/category |
| Page background opacity | Page canvas | Needed when the design relies on a tinted or dark surface |

### Visual-Type Sanity Checks

Use `references/chart-selection.md` to choose the visual type from the
analytical question first. Use this table only as a quick sanity check before
configuring the selected visual.

| Situation | Prefer | Avoid / switch when |
|---|---|---|
| Single KPI | `cardVisual` | Avoid charting one number unless trend/context is needed |
| Ranked categories | `barChart` horizontal | If >15 categories, add TopN/Other or use `tableEx` |
| Long category labels | `barChart` horizontal | Avoid vertical columns with rotated labels |
| Few time periods (≤6) | `columnChart` or cards | Avoid line chart with only 1–2 points |
| Continuous time trend (≥7 points) | `lineChart` | Avoid text month/quarter axes |
| Part-to-whole ≤5 slices | `donutChart` only if share is the question | Use sorted bar if comparison/ranking is the question |
| Part-to-whole >5 slices | sorted `barChart`, `treemap`, or 100% stacked bar | Avoid donut/pie |
| 2–3 same-unit measures | clustered bar/column or multi-line | Avoid mixing unlike units |
| 4+ series | small multiples or TopN + Other | Avoid spaghetti lines and overcrowded clustered bars |
| Need exact lookup/detail | `tableEx` / `pivotTable` | Avoid chart when precision is the main task |
| Correlation/clusters | `scatterChart` | Avoid dual-axis line/bar combos |
| Distribution/spread | histogram/binning or matrix/table fallback | Avoid showing only averages |
| Variance/deviation | waterfall or diverging bar treatment | Avoid arbitrary color without reference baseline |
| Geography or spatial distribution matters | `azureMap` | Use sorted `barChart` when geography is only a ranking category; avoid legacy `map` / `filledMap` |
| Operational status | cards + table + status icons/conditional formatting | Avoid decorative gauges |

### Bar & Column Charts(`barChart`, `clusteredBarChart`, `columnChart`, `clusteredColumnChart`)

**When to use**: Comparison across categories (bar = horizontal, column = vertical).
Horizontal bars are the DEFAULT for categorical comparison — they handle long labels
and >7 categories gracefully. **Minimum 6 categories** — with ≤5, use a donut
(part-to-whole) or stacked bar (composition) instead; bar charts look sparse and
the gradient pattern is ineffective at low cardinality.

**Design rules**:
- ALWAYS sort by value descending (biggest bar at top/left) via `sortDefinition`
  unless the axis has a natural order (Date column or numeric column —
  NOT text month/quarter names, which sort alphabetically in PBI)
- ALWAYS show data labels (`labels.show: true`, position: `OutsideEnd`) —
  eliminates the need for value axis gridlines
- When data labels are on, HIDE the value axis (`valueAxis.show: false`) and
  its gridlines (`gridlineShow: false`) — they're redundant
- **Value-gradient coloring** — single-measure bar charts render all bars in one
  color by default. Apply a `dataPoint.fill` FillRule gradient so the highest-value
  bar is the darkest and the lowest is the lightest — this reinforces the sort order
  visually. Use `linearGradient2` with `dataViewWildcard: { matchingOption: 0 }`
  selector. Pick colors from the theme's `dataColors` array: light tint for min,
  base color for max. **Each chart on the page gets a different hue** — assign
  palette colors by visual reading order and never reuse a hue already used by
  cards on the same page.
  - ⚠️ FillRule gradient color stops MUST use `Literal` hex values — `ThemeDataColor`
    silently renders black inside FillRule. Compute tint/shade from `dataColors[N]`.
  - Gradient works best when max/min value ratio is ≤10×. With extreme skew
    (one dominant outlier), consider a TopN filter or separate the outlier as a
    card callout.
- For clustered bars (AC vs PY), use contrasting palette colors (e.g.,
  amber + teal, not two blues)
- Category axis: 10pt, muted grey (`#605E5C`), no title, `innerPadding: 30`
  for comfortable spacing between bars
- Set `valueAxis.start` to 0 for bars — NEVER truncate the baseline
- Hide bar borders: `dataPoint.borderShow: false` for a cleaner look
- Hide legend when the category axis already shows the names (redundant);
  show when a separate Series field provides additional grouping

**Theme handles** (via `visualStyles.barChart["*"]`):
- labels show + position + fontSize
- valueAxis hidden + gridlines hidden
- categoryAxis styling + innerPadding
- dataPoint borderShow
- legend visibility

**Per-visual must set**: sort order, color/gradient treatment, title text

**Color assignment rule**: see `references/color.md` § Color Assignment Strategy.
Same measure as a card → match that card's color. Breakdown of a measure →
gradient of that color. Different measure → different hue.

### Line Charts (`lineChart`, `areaChart`)

**When to use**: Trend over continuous time. ONLY use if ≥3 time points — if
≤2 points, use cards or a column chart.

**Design rules**:
- **Smooth curves**: set `lineChartType: "smooth"` with `interpolationSmooth:
  "monotoneX"` — eliminates jagged segments, looks polished
- **Shaded area**: `areaShow: true`, `areaMatchStrokeColor: true` — fills
  below the line with a transparent version of the line color. Set area
  transparency to 80-85% via `dataPoint.transparency` so the area is subtle
- **Line width**: 2-3px for hero series (`strokeWidth: 2.5`), 1px for
  supporting context lines
- **Line color**: set via `dataPoint.defaultColor` — match the card that
  shows the same measure (see color assignment strategy in color.md)
- **Markers**: hide at ≥12 data points (clutter). Show at <12 for emphasis.
  Use `circle` shape, size 4-5px
- **Data labels**: sparse, neutral dark (#252423) — never the accent color.
  Set `labelDensity: 25` to show only key inflection points. Position
  `"Above"`. Do NOT enable label background unless the chart has busy gridlines
- For multi-line: limit to 3-4 series max. Beyond that, use small multiples.
  If you must show 5+, highlight one in accent and grey the rest
- **Reference line**: add a `y1AxisReferenceLine` for average/target — dashed
  gray (#A19F9D), width 1.5, with `dataLabelText: "ValueAndName"`. This
  gives context ("is this year above or below normal?")
- Date axis: auto-granularity is usually fine; override to month/quarter/year
  based on the data range. **Never use text month/quarter names** (e.g.,
  "January", "Q1") as the Category — they sort alphabetically. Use a Date
  column or numeric Month Number column instead.
- Gridlines: subtle dotted lines on the value axis help track values;
  category axis gridlines usually off

**Theme handles** (via `visualStyles.lineChart["*"]` in base.json):
- `lineStyles`: smooth interpolation, stroke width
- `categoryAxis` / `valueAxis`: label color, gridline style
- `labels`: show, sparse density
- `legend`: hidden for single-series

**Per-visual must set**: `dataPoint.defaultColor` (match card color),
`lineStyles.areaShow`, `y1AxisReferenceLine` (if adding average/target),
VCO title text

**Formatting objects**: `lineStyles` (strokeWidth, lineChartType,
interpolationSmooth, areaShow, areaMatchStrokeColor, showMarker,
markerShape, markerSize), `dataPoint` (defaultColor, transparency),
`labels` (show, labelPosition, labelDensity, fontSize, color),
`y1AxisReferenceLine` (show, value, style, lineColor, dataLabelShow),
`categoryAxis`, `valueAxis`, `legend`, `trend`, `forecast`, `zoom`

### Card Visuals (`cardVisual`)

**When to use**: Single headline metric. Always use `cardVisual` — `card` (legacy) is deprecated.

**Design rules**:
- Cards should show enough context to avoid bare-number ambiguity. Use
  `referenceLabel` or a concise contextual subtitle when it adds comparison,
  coverage, or time-window meaning: "+12% vs Q2 ▲" or "across 3 subjects".
- Give each card a distinct accent color from the palette — creates visual rhythm
- Left accent bar (`accentBar`) in the card's accent color adds polish
- No internal border (`outline.show: false`) — must be per-visual, NOT theme
- Clean label text — override `label.text` with human-readable name
- Cards should be compact — height ~11-15% of page height (88–112px on
  1280×720, 120px on 1920×1080). Too short (80px) cuts off labels; too
  tall (150px+) leaves dead space inside the card.
- Do not use a bare one-measure card as the page hero or largest visual. A card
  can be visually prominent only when it is part of a composite KPI treatment:
  value + delta/reference label, sparkline or threshold band, and a nearby
  explanation of why the number matters. Otherwise, move the card into a KPI
  strip/side callout and give the hero space to a trend, ranked driver chart,
  variance chart, map, table, or decomposition visual.

**Proportional sizing** (adapt to any canvas):
- Width: ~23% of page width per card (4 across with ~1.25% gutters)
- Height: ~11-15% of page height
- Example at 1280×720: w=296, h=96, gutters=16
- Example at 1920×1080: w=448, h=120, gutters=24
- Read `page.json` width/height first, then compute coordinates

**Theme handles** (via `visualStyles.cardVisual`):
- `value.bold: true` — callout numbers are bold by default
- `label.show: true` — category label visible by default

**Per-visual MUST set** (via `visual.objects` — theme cannot cascade these):
- `outline` (id: "default"): `show: false` — removes internal card border
- `accentBar` (id: "default"): `show: true`, `position: "Left"`, `width: 4D`,
  `color` = card's accent from palette
- `layout` (id: "default"): `topOuterMargin: 0L`, `bottomOuterMargin: 0L`,
  `leftOuterMargin: 0L`, `rightOuterMargin: 0L`, `paddingUniform: 0L` —
  eliminates the gap between card border and content so the accent bar
  sits flush against the edge
- `value` (id: "default"): `fontSize: 32D`, `fontColor` = accent color
- `label` (id: "default"): `text` = human-readable override, `fontSize: 10D`,
  `fontColor: #605E5C`
- `padding` (id: "default"): `Custom`, `leftMargin: 12L` (breathing room from
  accent bar), `rightMargin: 8L`, `topMargin: 0L`, `bottomMargin: 0L` — top/bottom
  must be 0 for the accent bar to sit flush (combined with layout outer margins)
- `cardCalloutArea`: `paddingUniform: 0L`

**VCO settings** (can be theme or per-visual):
- `padding`: `top: 0, bottom: 0, left: 0, right: 0` (zero — accent bar runs full height)
- `spacing`: `customizeSpacing: true`, `verticalSpacing: -6D` (tighten label-to-value)
- `border`: `show: true`, `color: #E8E8E8`, `radius: 8D` (outer rounded container)
- `background`: `show: true`, `color: #FFFFFF`, `transparency: 0D`
- `visualHeader`: `show: false`
- `title`: `show: false` (category label IS the title)

> ⚠️ **VCO mixing caveat**: If you set ANY VCO property per-visual
> (background, border, visualHeader), also set `padding` per-visual in the
> same `visualContainerObjects` block. Otherwise PBI may reset padding to
> its default (~5px) instead of inheriting from the theme.

**⚠️ Key limitation**: `outline`, `accentBar`, `layout`, and `padding` use
instance selectors (`id: "default"`) and do NOT cascade from theme `visualStyles`.
They must be set per-visual in `visual.objects`.

### Tables (`tableEx`)

**When to use**: Detail lookup — when the audience needs to read specific values,
not detect patterns. Use `tableEx` (modern), not `table` (legacy).

**Design rules**:
- ALWAYS set grow-to-fit: `columnHeaders.autoSizeColumnWidth: true`, `columnHeaders.columnAdjustment: "growToFit"`
- Column headers: bold, slightly darker background (#F3F2F1), 10-11pt
- Values: 10pt, right-align numeric columns, left-align text
- Alternating row colors: use `values.backColorPrimary` (#FFFFFF) and `backColorSecondary` (#FAFAFA) — subtle, NOT zebra stripes. MUST also set `fontColorPrimary` and `fontColorSecondary` or text goes invisible
- Sort by the most interesting column descending (highest enrollment, worst performance, etc.)
- Conditional formatting: add data bars (`dataBar` CF) on the primary numeric column — makes magnitudes scannable without reading numbers
- If table has a status/result column, add icon set CF (`iconRule`) — ✓/✗ or traffic light icons
- Limit visible rows: if the underlying data has >20 rows, add a page-level or visual-level Top N filter
- Total row: show for additive measures (Sum of Revenue), hide for non-additive (Average, %)

**Formatting objects**: `columnHeaders` (autoSizeColumnWidth, columnAdjustment, bold, backColor, fontColor, fontSize), `values` (fontSize, fontColor, backColorPrimary, backColorSecondary, fontColorPrimary, fontColorSecondary), `total` (show, bold, backColor), `grid` (gridlineColor, rowPadding)

**Sizing**:
- Minimum height: **240px** — shows ~8 data rows with styled header + banding
- Preferred: 280–320px for detail-heavy tables
- If the table is the bottom visual, expand height to fill remaining canvas
  rather than leaving whitespace below
- Full-width tables should span 1232px (12 columns) — if slicers are inline
  with the title, the table gets the full width

### Matrix (`pivotTable`)

**When to use**: Cross-tabulation with row and column grouping. Use `pivotTable`
(modern), not `matrix` (legacy).

**Design rules**:
- Same grow-to-fit and alternating rows as tableEx
- Row headers: bold, left-aligned
- Column headers: bold, centered
- Subtotals: show for the dimension the audience cares about; position at top/bottom based on reading convention
- Stepped layout for hierarchies — indent child rows under parent
- Conditional formatting: background color gradient (`fillRule` CF) on the values cells — makes the heatmap pattern visible
- Expand/collapse: set `expansionStates` for the default view — usually expand the first level, collapse deeper levels

**Formatting objects**: Same as tableEx plus `rowHeaders` (bold, fontSize), `subTotals` (show, bold, backColor), `columnSubtotals`, `columnHeaders`

### Slicers

2-3 slicers per page maximum. Pick dimensions that control the most visuals
on the page — if 3 charts use Weight Class and 2 use Year, those are your
slicers. Avoid slicing by the same field on a chart's category axis.

**Type selection:**

| Data | Type | `visualType` |
|------|------|-------------|
| Categorical ≤10 values | Button/tile | `advancedSlicerVisual` |
| Categorical 10-50 | List | `listSlicer` |
| Categorical >50 | Dropdown | `slicer` (mode: Dropdown) |
| Annual or period selector | Dropdown or tile | `slicer` (mode: Dropdown) or `advancedSlicerVisual` |
| True date range | Date picker | `slicer` (mode: Between) |
| Hierarchy | Tree | `listSlicer` |

### Temporal slicer decision matrix

Choose the slicer from the page question and data grain, not from the mere
presence of a full-date column.

| Signal | Recommended slicer | Rationale |
|---|---|---|
| Executive page, annual grain, ≤10 years | Year dropdown or year tile | Compact, reliable, and matches scan-first behavior |
| User needs to compare discrete periods (e.g., 2020 vs 2023) | Year/month dropdown with multi-select | `Between` implies continuous ranges, not discrete comparisons |
| Month/quarter reporting with 12-36 periods | Period dropdown or relative date | Keeps the header compact while exposing current period |
| Analyst needs arbitrary day/month range exploration | Full-date `Between` | Range selection is the task |
| Date column is an integer key/text key or did not render as a date picker | Dropdown on Year/Period | `Between` requires a renderable Date/DateTime field |
| Many dates but no need for arbitrary ranges | Relative date or period dropdown | Avoids oversized controls for low-value precision |

**Placement** (choose by slicer count — see `references/layout.md`):

| Count | Position | Layout impact |
|---|---|---|
| 1–3 | Inline with title row (right side) | No width penalty — full chart width preserved |
| 4+ | Vertical filter rail (200–240px left) | F-pattern, justified when slicers fill the column |

Never use a vertical rail for ≤3 slicers — it wastes 70%+ of the
column height as dead space and unnecessarily narrows all content.
Never place slicers in the top-left corner — that's for the page title.

#### Dropdown Slicer (`slicer`, mode: Dropdown)

The default choice — compact, searchable, handles any cardinality.

**Sizing** (proportional to canvas):
- Height: follow the slicer sizing guidance in `powerbi-report-authoring` exactly; default dropdown is
  `60 + top_padding + bottom_padding`, snapped to the 8px grid. Do not shrink
  to 48px or 8pt to make a tight layout work.
- Width: ~12% of page width (about 230px on FHD, 160px on 1280×720). Use ~10%
  for short labels like "Year" or "Stance".

**Two styling variants:**

| | **Standard (fill)** | **Light (no fill)** |
|---|---|---|
| Background | White `#FFFFFF` | None |
| Border | `#E8E8E8`, radius=8 | `#E8E8E8`, radius=8 |
| Use for | Top strip, inline with cards | Sidebar rail |

Both share border radius=8 and color=#E8E8E8, matching card visuals.

**Theme handles** (via `visualStyles.slicer` in base.json):
- `header`: body-lg font (10pt Semibold), outlineStyle=0 (no header border)
- `items`: body font (9pt Regular), outlineStyle=0, padding=2

**Per-visual must set:**
- `header.text` — display name ("Weight Class" not "weight_class_name")
- `data.mode` — "Dropdown" (or Basic, Between, etc.)
- VCO `background`, `border`, `padding` — for fill vs light variant

#### Date Between Slicer (`slicer`, mode: Between)

Use for temporal filtering only when arbitrary date-range exploration is part
of the page question and the bound field is a renderable Date/DateTime column.
For executive dashboards with yearly or quarterly grain, prefer a compact
Year/Period dropdown or tile.

**Sizing**:
- Inline with title: w=216; height is `60 + top_padding + bottom_padding`,
  snapped to the 8px grid. Dates render side-by-side at this width.
- Vertical rail: w=200; height is `84 + top_padding + bottom_padding` because
  dates stack vertically at this width.
- Authoring must use the exact slicer sizing formula from `powerbi-report-authoring`
  after theme/VCO padding is known.

**Styling variants**: same Fill and Light as Dropdown Slicer (see table above).

**Theme handles** (via `visualStyles.slicer` in base.json — shared with dropdown):
- `header`: 10pt Semibold, outlineStyle=0
- `date`: 9pt font for date picker text

**Per-visual must set:**
- `data.mode` — "Between"
- `header.text` — "Date Range" or similar
- VCO `background`, `border`, `padding` — for fill vs light variant

**When to prefer Dropdown/Tile over Between:**
- Executive or scan-first page with annual/quarterly grain
- Integer/text year-month key or date picker did not render
- User expects multi-select (e.g., compare 2020 vs 2023)
- ≤30 distinct values where a list is scannable

### Textbox (page titles, annotations)

**When to use**: Descriptive titles, section labels, body prose in narrative, footnotes.

**Design rules**:
- Page title (H1): 20pt SemiBold, dark navy (#252423), top-left, width ≥600px
- Section dividers: use `shape` (rectangle) not textbox — textbox has a ~24px minimum height
- Annotations: 10-11pt Regular, positioned near the chart element they reference
- Rich text: use `paragraphs[].textRuns[]` for inline bold, color, sizing
- Dynamic text: bind to a measure for data-driven titles ("Revenue is {[Revenue]} this quarter")
- Footer text: 8-9pt, muted grey (#605E5C), bottom of page, source + refresh timestamp

**⚠️ Textbox height — avoid scrollbar**: A scrollbar appears when the
textbox height is too small for the font size. Use this formula:

```text
h_min = max(18, ⌈fontSize_pt × 25/16⌉) + padding_top + padding_bottom
```

| Font Size | Min Height (no padding) | With 8+8 padding |
|-----------|------------------------|------------------|
| 10pt | 18px | 34px |
| 14pt | 22px | 38px |
| 20pt | 32px | 48px |
| 24pt | 38px | 54px |
| 28pt | 44px | 60px |

- Floor of 18px regardless of font size
- VCO border adds ZERO pixels (rendered outside bounds)
- Font family and bold/regular do NOT change the threshold
- Set VCO padding to 0 if you want the tightest possible fit

**PBIR note**: `paragraphs` is a native JSON array — never stringify it

### Shape (`shape`)

**When to use**: Decorative dividers, section backgrounds, annotation arrows, call-out boxes.

**Design rules**:
- Divider line: rectangle, h=2, w=full page width, fill=#E0E0E0 — separates page sections
- Section background: rounded rectangle, low opacity fill (#F5F5F5), z-ordered BEHIND the visuals it groups
- Annotation arrow: use arrowhead geometry, point at the chart element being called out
- Decorative shapes are acceptable when they support the report's tone; keep data-bearing areas legible
- Shapes don't have a 24px minimum like textboxes — use them for thin dividers

**Formatting notes**: Shape visual uses `id` selectors (not metadata/scope). `fill`, `outline`, `text`, `shadow`, `glow` MUST have `"selector": { "id": "default" }`. The `shape` object (tileShape/geometry) needs NO selector.

### Scatter (`scatterChart`)

**When to use**: Relationship between two continuous variables. Correlation, clusters, outliers.

**Design rules**:
- X and Y axes: always labeled with measure name and unit
- Bubble size role: use sparingly (3rd variable) — area encodes poorly (Stevens' law)
- Data labels: show on outlier points only, not all (visual clutter)
- Quadrant reference lines: add horizontal + vertical `referenceLine` to divide into performance quadrants
- Color by category: use legend for identity, not for magnitude
- Play axis: enable for time-lapse animation when a date dimension exists

### Waterfall (`waterfallChart`)

**When to use**: Sequential additive contribution — how does a total build up from
components? Variance bridges (start → adjustments → end).

**Design rules**:
- Use the theme's sentiment colors for increase (good=#1AAB40) and decrease (bad=#D64554)
- Total bar: distinct color (neutral blue #0072B2 or grey)
- Data labels: always on — the reader needs the exact numbers
- Category labels: name each step clearly
- Connector lines: show to maintain the visual bridge

### Treemap (`treemap`)

**When to use**: Part-to-whole when there are >10 categories. NOT for comparison (use bar chart).

**Design rules**:
- Limit to 1 grouping level unless the hierarchy is the point
- Color: use the categorical palette so each rectangle is distinct
- Data labels: show category name + value/percentage inside each rectangle
- Size encodes magnitude — the biggest rectangle is the dominant category

### Conditional Formatting — MANDATORY

Use conditional formatting when it reinforces the page's main comparison,
threshold, or exception pattern.

| Technique            | Use on                 | Effect                                                                                                                                                |
| -------------------- | ---------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Per-series color** | Bar/column charts      | Each category bar gets a distinct color when category identity matters |
| **Threshold color**  | KPI cards              | Background or font color changes based on good/neutral/bad thresholds                                                                                 |
| **Data bars**        | Table numeric columns  | In-cell horizontal bars show magnitude at a glance                                                                                                    |
| **Icon sets**        | Table status columns   | Pass/fail or good/warn/bad status becomes scannable                                                                                                   |
| **Color gradient**   | Matrix cells, heatmaps | Min→mid→max color scale reveals patterns                                                                                                             |
| **Highlight + grey** | Any chart              | One series in accent color, rest in #BDBDBD — draws the eye to the insight                                                                            |

**The "all bars same color" problem**: single-series bar charts can render
every bar with the same default color. If category contrast matters, specify
per-category colors or a sequential gradient in the design brief and let
authoring handle the mechanics.
